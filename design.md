## 一行サマリ

**ブラウザを真の OS として正面から扱い、WASI を共通契約として、過去のバイナリも未来のアプリも、どんなハードウェア上でも同じ速度で走らせるための実行基盤を作る。**

## 中心命題: ブラウザこそが真の OS である

### この命題が意味すること

従来の常識:

> ブラウザは Windows / macOS / Linux の上で動くアプリケーションである。

本プロジェクトの命題:

> **ブラウザこそが、ユーザーから見て真の OS である。**
> Windows / macOS / Linux は、ブラウザに対して「ハードウェア層」と「ネットワーク経由でアクセスする外部資源」の役割しか持たない、薄い土台にすぎない。

この視点の転換は、修辞ではなく具体的な設計判断を伴う。

- **ブラウザの API 群** (`WebAssembly`, OPFS, Service Worker, WebUSB, Cache API, File System Access) は、本プロジェクトにおける **真のシステムコール** である。これらが POSIX における `open`, `read`, `write`, `mmap` に相当する。
- **ホスト OS のファイルシステム** は、ブラウザから見たときの **外部ネットワーク資源** である。ローカルディスク、NAS、クラウドストレージは、すべて「リモートマウント」として等価に扱う。物理的距離が違うだけで、インターフェースは統一する。
- **ホスト OS のプロセスモデル** には触らない。ブラウザは自分の中に独自のプロセス概念 (タブ、Worker、Wasm インスタンス) を持つ。これらが本プロジェクトにおけるプロセスである。
- **ホスト OS のドライバ層** は、ブラウザの WebUSB / WebHID / WebSerial / WebBluetooth が代替する。ブラウザに見えるデバイスだけが、本プロジェクトにとっての「実在するデバイス」である。

### なぜこの命題が成立するのか

歴史的に、OS の役割は以下の四つだった:

1. ハードウェアの抽象化
1. プログラムの実行管理
1. リソース (メモリ、CPU、ストレージ) の配分
1. プログラム間の隔離とセキュリティ

現代のブラウザは、これらすべてを既に提供している:

1. **ハードウェア抽象化**: WebGL/WebGPU、WebUSB、WebAudio、WebRTC が、CPU・GPU・USB・音声・ネットワークを抽象化する。
1. **プログラム実行**: V8 / SpiderMonkey / JavaScriptCore が、JavaScript と WebAssembly のための仮想機械を提供する。
1. **リソース配分**: タブごとのメモリ制限、CPU スロットリング、Storage Quota がリソースを管理する。
1. **隔離とセキュリティ**: Origin による分離、サンドボックス、CSP、Permissions API が、Linux のプロセス分離に匹敵する保証を提供している。

ブラウザは既に OS の機能を概ね持っている。「ブラウザは OS ではない」という言い方は、慣習にすぎない。

### この命題から何が変わるか

「ブラウザは OS ではない」という前提で設計すると、以下のような不要な迂回が生じる:

- ローカルファイルへのアクセスを「ブラウザの制限を回避する手段」として実装する
- ファイルシステムを「本物 (ホスト OS) と仮想 (ブラウザ内) の二系統」として扱う
- パフォーマンスのギャップを「ブラウザだから仕方ない」と諦める

「ブラウザは OS である」という前提で設計すると、これらが解消する:

- ローカルファイルは「外部ネットワーク資源」として、NAS やクラウドと同じ扱い
- ファイルシステムは一つだけ。ブラウザの提供する OPFS と Cache API がプライマリで、外部資源は遠いマウントポイント
- パフォーマンスは「OS のシステムコールとして当然の水準」を要求する

## 背景と動機

### 出発点の観察

現代のブラウザは、すでに事実上の OS である。テキスト編集、コーディング、コミュニケーション、画像編集、3D まで、日常作業の大半をブラウザ内で完結できる。残っている不自由 (ローカルファイル、低遅延デバイス、メモリ上限、バックグラウンド処理) は、ブラウザの本質的な制約ではなく、「ブラウザは OS ではない」という前提から派生した周辺的な制約である。

この前提を捨てれば、不自由のかなりの部分は消える可能性がある。

### コアな問題意識

OS、CPU アーキテクチャ、プログラミング言語、配布形式 — これらが互いに分断されている現状は、技術的必然ではなく歴史の偶然である。

- 同じソフトウェアを Windows 版、Mac 版、Linux 版、ARM 版、x86 版と別々にビルドする状況
- 特定のハードウェアでしか動かない古いソフトウェアが死蔵される現象

これらは、共通の中間層が欠けていることに起因する。WebAssembly と WASI は、この欠けたピースを埋めうる規格として急速に成熟しつつある。本プロジェクトはこの規格を、ブラウザという既に普及した実行環境の中で実体化する。

### なぜ今やるべきか

- **WASI の成熟**: WASI Preview 2 が安定化し、コンポーネントモデルが実装フェーズに入った (2024〜2026)。
- **ブラウザの普及**: 世界人口の半分以上が日常的にブラウザを使う。
- **AI エージェントの時代**: ファイルシステムにアクセスでき、永続化でき、サンドボックス化された実行環境は、AI 時代のインフラとして要求が高まっている。
- **ハードウェアの多極化**: x86 一強の時代は終わり、ARM、RISC-V、各種アクセラレータが乱立する。共通の実行抽象が以前にも増して必要。

## 設計思想 — 五つの原理

このプロジェクトを貫く五つの原理。すべての設計判断はここから演繹される。

### 原理 1: ブラウザこそが OS である (Browser as the True OS)

ブラウザを「OS の上で動くアプリ」ではなく、「ユーザーにとっての真の OS」として正面から扱う。
ホスト OS は、ブラウザという OS から見た「ハードウェア層」と「外部ネットワーク資源を持つ遠隔サーバー」にすぎない。

**帰結**:

- ブラウザの API を「真のシステムコール」として扱う。`WebAssembly.compile()` は `mmap(PROT_EXEC)`、OPFS は POSIX FS、Service Worker は init プロセスに相当する。
- ホスト OS のファイルシステムは、`remote://local-disk/` というリモートマウントとして抽象化する。ローカル、NAS、クラウドの差はレイテンシのみ。
- ブラウザに見えないものは、本プロジェクトにとって「存在しない」。ホスト OS が掴んでいるデバイスや特権 API には触れない。

### 原理 2: 規格を中央に置く (Standards as Sovereignty)

主権はハードウェアでも特定企業でもなく、**WASI という開かれた規格**に宿る。
誰かが独占できる中央サーバーや特定 CPU メーカーに依存しない。

**帰結**:

- WASI 仕様への準拠を最優先する。独自拡張は最小限に。
- Bytecode Alliance の議論を追跡し、必要なら参加する。
- 「自分たちだけが速い」より「規格通りに書けば誰でも速い」を優先。

### 原理 3: ハードウェアを選ばない (Hardware Agnosticism)

x86 でも ARM でも RISC-V でも、最新でも 10 年前でも、同じバイナリが動く。
ユーザーは新しいハードウェアを買わなくていい。

**帰結**:

- 第一の実装ターゲットは「既存のホスト OS 上のブラウザ」。専用ハードウェアは後回し。
- WASI ネイティブシリコンは「ゴール」であって「前提」ではない。
- どの環境でも、同じ UX を提供する (速度差は許容、機能差は許さない)。

### 原理 4: 過去を救い、未来を建てる (Continuity & Progress)

新しい規格に乗り換えるためのコストをユーザーに負わせない。
古い x86 バイナリも、未来の WASI ネイティブアプリも、同じプラットフォームで地続きに動く。

**帰結**:

- x86 / ARM エミュレーション (CheerpX 等) を第一級市民として扱う。
- WASI ネイティブのユーザーランドを並行して育てる。
- ユーザーは「これは互換レイヤーで動いている」「これはネイティブで動いている」を意識しなくていい。

### 原理 5: 既存の上に乗る (Parasitic Growth)

新しい OS は普及しない、歴史が証明している。
だから戦略は逆 — みんながすでにやっていること (ブラウザを開く儀式) に寄生する。

**帰結**:

- 第一のデリバリ形態は「普通のブラウザで動く Web アプリ」。新規インストール不要。
- 「ホスト OS 起動直後にこれが立ち上がる」を最終形態として設計するが、最初はその一歩手前で止める。
- ユーザー行動を 1mm も変えずに、実質的な体験を別物にする。

### 五原理の相互関係

- **原理 1 が中心** にあり、他の四つはそこから派生する具体戦略である。
- **原理 1 と 2** が組み合わさって、「ブラウザという OS の中身は、特定企業の独自実装ではなく、開かれた規格 (WASI) で記述される」という設計を生む。
- **原理 1 と 3** が、「ブラウザという OS は、下のハードウェアを問わない」という性質を保証する。
- **原理 1 と 4** が、「過去の Linux バイナリも、未来の WASI バイナリも、同じブラウザ OS 内で動く」という統合をもたらす。
- **原理 1 と 5** が、「新しい OS をインストールする必要はない、既にあるブラウザ OS を活性化するだけだ」というメッセージになる。

```
├─────────────────────────────────────────────────────────────┤

├─────────────────────────────────────────────────────────────┤

│   - WebAssembly API → 実行・JIT                              │
│   - OPFS / Cache API → プライマリ FS                          │

│  ブラウザエンジン (V8 / SpiderMonkey / JavaScriptCore)         │
├─────────────────────────────────────────────────────────────┤
│  以下、ブラウザ OS から見た「外部世界」                           │
│  ┌──────────────────────────────────────────────────────┐    │
│  │ ホスト OS (Linux / macOS / Windows / うっすい OS)       │    │
│  │  ↑ ブラウザ OS から見ればこれは「ハードウェアと        │    │
│  │     外部ネットワーク資源を持つ遠隔サーバー」              │    │
│  └──────────────────────────────────────────────────────┘    │
│  ┌──────────────────────────────────────────────────────┐    │
│  │ 物理ハードウェア (x86 / ARM / RISC-V / WASI シリコン)    │    │
│  └──────────────────────────────────────────────────────┘    │
└─────────────────────────────────────────────────────────────┘
```

### キーコンセプト

- **タブ ≒ プロセス**: ブラウザのタブは、ブラウザ OS におけるプロセスの一形態である。ただしタブは POSIX プロセスとライフサイクルが異なる (ユーザーが手動で閉じる、OOM killer に殺される、復元される) ため、完全な等価ではない。シェルのバックグラウンドジョブやパイプは、1 タブ内の複数 Wasm インスタンスで実装する方が現実的な可能性がある。
- **タブを閉じる ≒ サスペンド**: メモリスナップショットと FS 差分を永続化層に書き出す。
- **タブを開く = レジューム**: キャッシュからスナップショットを復元、瞬時に「前の続き」へ。
- **Service Worker = init**: タブが閉じても生存し、ストレージ層と通信ハブを担う。
- **AI 砂場プロセス**: メイン作業タブとは別プロセスで動く AI エージェント専用サンドボックス。プロセスレベルで分離。

## 構成要素と作るべきもの

優先度別に整理する。
**[★] 初期プロトタイプ必須**、**[☆] 中期で必要**、**[◇] 長期 / オプション**。

### A. シェルとユーザーランド

- **[★] WASI 対応ターミナル UI**
  xterm.js ベース。標準入出力を Wasm の stdio に接続する薄いラッパー。
- **[★] WASI 版 bash (または互換シェル)**
  既存の Wasm 移植版を活用、または BusyBox 風の軽量シェルを自作。
  目的は「ブラウザ OS のファイルシステムと完全に対話できる最小単位」。
- **[☆] WASI 版 coreutils**
  ls, cat, cp, mv, grep, find など。多くは既に Wasm 移植が存在するので、それを束ねる。
- **[☆] WASI 版エディタ**
  vim or nano の Wasm 移植。
- **[◇] WASI 版コンパイラ / ランタイム**
  Python, Node, gcc などを WASI 上で動かす。NAS にキャッシュして再利用。

### B. ファイルシステム — 一系統に統一する

ブラウザ OS のファイルシステムは一つだけ。プライマリは OPFS / Cache API、それ以外はすべてリモートマウント。

ただし OPFS はブラウザのストレージ規約に従い、容量圧迫時に削除される可能性がある。実態としては OPFS は L2 キャッシュに近く、真のプライマリストレージはリモートマウント先 (NAS / クラウド / `remote://local/`) という構造になりやすい。この点はトレードオフとして認める。

- **[★] WASI ↔ OPFS ブリッジ**
  WASI の `fd_read` / `fd_write` / `path_open` を、OPFS の SyncAccessHandle に最短経路でマップ。Web Worker 内で動かして JS メインスレッドを経由しない。
- **[★] リモートマウント抽象**
  `remote://` という統一プロトコルの下に、外部資源を吊り下げる:
  - `remote://local/` → File System Access API 経由でホスト OS のフォルダ
  - `remote://nas/` → WebDAV / SFTP over WebSocket
  - `remote://s3/` → S3 互換オブジェクトストレージ
  - `remote://cloud/` → Google Drive、Dropbox 等
- **[☆] スナップショット機構**
  Wasm 線形メモリと FS 差分を効率よくシリアライズ / デシリアライズ。Copy-on-Write で差分のみ保存。
- **[◇] 分散ストレージ統合**
  IPFS、独自プロトコルなどを `remote://` 配下に追加。

### C. ネットワーク抽象

- **[★] Fetch ベースの HTTP クライアント**
  WASI から呼べる形に薄くラップ。多くのユースケースはこれで足りる。
- **[☆] WebSocket / WebTransport ゲートウェイクライアント**
  NAS / エッジサーバー側に立てた中継器を経由して、任意のプロトコルを透過的に使う。
- **[◇] 自前の中継サーバー実装**
  WebSocket → 任意 TCP/UDP の変換器。NAS に同梱できる軽量バイナリ。

### D. 互換レイヤー

- **[★] WebVM / CheerpX の埋め込み**
  既存の x86 Linux 互換レイヤーを最初のプロトタイプに同梱。原理 4 の実証。
- **[☆] 翻訳結果のキャッシュ機構**
  CheerpX の x86 → Wasm 翻訳結果をブラウザ側で永続化、再利用する仕組み。
- **[◇] AOT 翻訳パイプライン**
  サーバー側で x86 バイナリを事前に WASI へ翻訳して CDN 配信。Rosetta 2 的アプローチの WASI 版。

### E. ブラウザ OS のシステムコール抽象化層

ここがプロジェクトの心臓部。ブラウザ API を WASI 互換のシステムコールとして再パッケージする層。

- **[★] WASI Preview 1 互換シム**
  `@bjorn3/browser_wasi_shim` のような既存実装をベースに改造。
- **[☆] WASIX / WASI Preview 2 サポート**
  `proc_fork`、コンポーネントモデルを利用するアプリ向け。
- **[☆] 動的 JIT 機能の提供**
  「Wasm の中から `WebAssembly.compile()` を呼ぶ」インターフェースを WASI 拡張として提供。Python インタープリタ等が JIT を行えるようにする。

### F. デリバリと永続化

- **[★] Service Worker による Wasm キャッシュと先読み**
  初回起動を 3 秒以下、2 回目以降を 1 秒以下にするための核心。
- **[★] スナップショット起動**
  bash が起動した状態のメモリイメージを保存しておき、復元するだけで 0.5 秒以下で「使える状態」にする。
- **[☆] 状態同期プロトコル**
  複数デバイスで同じ環境にアクセスしたときの整合性。最初は「最後の書き込みが勝つ」で十分。

### G. ホスト環境

ブラウザ OS が走る土台。原理 1 によれば、これらは「外部世界」であり、薄ければ薄いほど良い。

- **[★] 普通のブラウザ上で動く Web アプリ版**
  最優先。Chrome, Firefox で同等に動くこと。原理 5 の核心。
  
  Safari は WebUSB / File System Access API などが未実装で、機能制限版になる。iOS Safari は特に制約が大きく、第一フェーズではターゲット外として明示する。
- **[☆] Electron / Tauri 版**
  ホスト OS のファイルシステムへのフルアクセスと、ローカル中継サーバーの内蔵。普通のブラウザ版とコードベースを共有する。

## POSIX 互換性: 達成可能な範囲と原理的な限界

WASI を POSIX 互換に近づければ近づけるほど、既存ソフトウェアが動く。
しかし完全な POSIX 互換は、Wasm のサンドボックス設計と原理的に衝突する。
これは Wasm の欠陥ではなく、Wasm が安全性のために選んだ設計上のトレードオフである。

### 原理的に厳しい部分 (永遠に完全互換にならない)

- **`fork()` の任意地点での意味論**: メモリ空間の即時複製を Wasm 線形メモリで完全再現することはできない。実行コンテキスト (PC、スタック) の取得・復元も現行仕様では不可。
- **`mmap()` の高度な使い方**: ファイルの直接マップ、共有メモリ、`MAP_FIXED`、実行可能ページの動的生成。最後の項目は **Wasm 内で Wasm を JIT コンパイルできない** という形で現れる。
- **シグナルとプリエンプティブな割り込み**: `SIGINT`、`SIGSEGV` をハンドラで非同期に捕まえる仕組み。Wasm のセキュリティモデルが拒否する。
- **直接的な I/O ポート、`ioctl`、生ソケット**: サンドボックスが意図的に提供しない。
- **POSIX 完全準拠の IPC**: ファイル記述子の渡し合い (`SCM_RIGHTS`)、System V IPC の細部。

### 結果として動かないもの (スコープ外と認める)

- ネイティブ JIT に強く依存する高性能ランタイム (LuaJIT、PyPy、JVM HotSpot のフルスピード版)
- 古典的 fork ベースの並列プログラム (現代のソフトの大半は `posix_spawn` で代替できるので、影響範囲は限定的)
- リアルタイム性が厳しいシステム (音楽制作の超低遅延、ロボット制御、医療機器)
- 低レベルなデバイス操作ツール (組み込み開発、システム管理の一部)

これらは将来の Wasm 拡張で部分解決する可能性と、永遠に解決しない可能性の両方を持つ。プロジェクトとしてはスコープから明示的に外し、必要な人は別の手段を使うことを推奨する姿勢を取る。

### 達成可能な目標: POSIX の 90% 互換

現代のソフトウェアの 9 割以上は POSIX のありふれた使い方しかしていない:

- ファイルを開いて読み書きする
- ネットワークソケットで通信する
- 子プロセスを起動する (`posix_spawn` 相当)
- マルチスレッドで並列計算する
- ユーザーからの入力を受け取る

これらは WASI / WASIX で十分カバーできる。

## Wasm 制約への創造的な解法

POSIX 完全互換が原理的に厳しいことを認めた上で、原理 1 (ブラウザこそが OS) の視点に立つと、いくつかの制約は別の見え方をする。Wasm 単体では解決できないが、ブラウザ OS の他のシステムコール (WebAssembly API、Service Worker 等) と組み合わせれば解決できる、という構造である。

### 解法 1: タブ複製とスナップショットによる「セーブポイント fork」

POSIX の `fork()` は任意地点での即時複製を要求するが、本プロジェクトでは **明示的なセーブポイントでの fork** に意味論を変更する。

**設計**:

- Wasm の線形メモリと制御状態を「凍結」できる地点をプログラム側が定義する
- その地点でタブ複製または新インスタンス起動を行い、凍結イメージから再開する
- bash のような「コマンド実行直前にフォークするシェル」には十分な機能性

**ブラウザ OS のシステムコール対応**:

- 凍結 → OPFS への書き込み (`fd_write`)
- 新インスタンス起動 → Service Worker 経由でのタブ複製、または同タブ内 Worker 起動
- 復元 → OPFS からの読み出し + `WebAssembly.instantiate()`

**根拠**: WASIX の `proc_fork`、Erlang のプロセスモデル、Smalltalk のイメージベース実行が、概念的に同じ方向を指す。本プロジェクトはこれをブラウザのタブ複製機構と統合する。

### 解法 2: 一系統のファイルシステムと外部資源のリモートマウント

これは制約への対処というより、原理 1 を徹底することで「制約」が消える例である。

ブラウザに二つのファイルシステム (仮想 FS と本物の FS) があると考えるから、複雑になる。ブラウザ OS のファイルシステムは一つ (OPFS) で、それ以外は外部ネットワーク資源、と整理すれば、ファイルシステム問題はネットワーク問題に還元される。

**設計**:

- **プライマリ FS**: OPFS。ブラウザ OS 内部の高速ストレージ。Service Worker と連動して永続化。
- **リモートマウント**: `remote://` 配下に外部資源を吊り下げる。
  - `remote://local/` → File System Access API
  - `remote://nas/` → WebDAV over WebSocket
  - `remote://cloud/` → 各種クラウド API

**実装上の注意**:

- ブラウザのストレージは恒久ではない (容量圧迫時に削除される可能性)。`navigator.storage.persist()` で永続化を要求しても、ユーザーがサイトデータを削除すれば消える。
- 真に重要なデータは、リモートマウント先 (NAS や `remote://local/`) に複製する設計を組み込む。OPFS はあくまで作業用キャッシュ。
- 定期的な同期をバックアップとして組み込む。

**意義**: ローカル、NAS、クラウドを統一プロトコルで扱うことで、デバイス依存性が消える。原理 3 (ハードウェア独立) を強化する。

### 解法 3: ネイティブ JIT の「ホスト依頼」モデル

Wasm 内で実行可能ページを動的生成できないという制約は、原理 1 に立つと別の意味を持つ。

ブラウザ OS においては、`WebAssembly.compile()` こそが「実行可能メモリを動的に確保する」ためのシステムコールである。Linux でいう `mmap(PROT_EXEC)` に相当する。Wasm モジュール内部でこれを呼べないのは、ユーザーランドプロセスが直接 `mmap` を実装できないのと同じ — OS (ブラウザ) に依頼すれば良いのである。

**設計**:

- インタープリタ (Python インタープリタ等) が、実行時にホットコードを検出する
- ホットコードを Wasm バイトコードに動的生成する
- ブラウザ OS のシステムコール `WebAssembly.compile()` を呼んで JIT を依頼する
- 生成された Wasm モジュールから関数をインポートして高速実行

**性能の見積もり (要検証)**: ネイティブ JIT (PyPy 等) の数十パーセント程度を期待。ただし `postMessage` 経由の通信オーバーヘッドや、`WebAssembly.compile()` のレイテンシ次第で大きく変動する。実測値は未取得。

**先行例**: Truffle / GraalVM の Wasm バックエンド、`wasm-jit` 系の実験的プロジェクト。

### 解法 4: USB Type-C と WebUSB によるデバイス支配 — 二段階達成

すべての物理デバイスをブラウザ OS から操作する目標は、現代ブラウザでは部分達成、うっすい OS では完全に近づくという二段階で整理する。

**現代ブラウザでの達成範囲** (部分的):

- WebUSB、WebHID、WebSerial、WebMIDI、WebBluetooth、WebNFC でホビー用途と特定業務用途は十分カバー
- 限界: ホスト OS が掴んだデバイス (内蔵ストレージ、ネットワーク、内蔵カメラ) には触れない
- 限界: 低レイテンシ要求、ベンダーの WebUSB 対応の有無、iOS Safari の未実装

**うっすい OS での達成範囲** (ほぼ完全):

- ホスト OS が薄ければ薄いほど、OS による「デバイスの掴み」が減る
- 極限の薄い OS では、ブラウザがほぼ全デバイスに直接話せる
- WASI シリコンや WASI ネイティブカーネルでは、デバイスドライバ層も WASI で抽象化可能

**戦略**: 第一フェーズでは現代ブラウザでの WebUSB / WebHID 統合を実装し、限界を明示する。専用機フェーズでデバイス支配の完成形を目指す。

## 既知の課題と未解決問題

正直に書く。これらは解決されていない。

### 技術的課題

- **マルチコア活用**: WASI スレッドはまだ未成熟。SharedArrayBuffer の COOP/COEP 制約も重い。最初はシングルスレッドで諦める。
- **GPU / アクセラレータ**: WebGPU は描画寄り。汎用計算の WASI 抽象は未確立。AI 推論などは外部サービスに逃がす。
- **メモリ上限**: ブラウザの 4GB 壁は越えられない。重い処理はエッジに送る設計で逃げる。
- **リアルタイム性**: 音楽制作、ロボット制御などの厳しい要求はスコープ外とする。
- **永続化のコンシステンシー**: 複数デバイス同時操作の整合性。最初は「最後の書き込みが勝つ」で逃げる。
- **ストレージの非永続性**: ブラウザのキャッシュは容量圧迫時に削除される可能性がある。重要データは外部マウント先に複製する設計が必須。実態として OPFS は L2 キャッシュ、リモートマウント先がプライマリストレージ、という構造になりやすい。
- **タブとプロセスの不一致**: 「タブ = プロセス」の比喩は便利だが、タブのライフサイクル (手動クローズ、OOM 時の挙動、復元) は POSIX プロセスと異なる。シェルのバックグラウンドジョブやパイプは、別の実装戦略が必要かもしれない。

### 戦略的課題

- **キラーアプリが定まっていない**: 「便利」だけでは普及しない。最初に響くユースケースを見つけ続ける必要がある。ペルソナ 3 (古いソフトウェアの永久保存) を最初の楔として仮置きしているが、実際に響くかは未検証。
- **エコシステムの育成**: 配布レジストリ、署名、バージョン管理など、パッケージ管理の戦略を決めていない。
- **標準化への関与**: Bytecode Alliance への参加には時間と政治的コストがかかる。
- **原理 1 と原理 5 の緊張**: ユーザー視点では「いつものブラウザでいつものWebアプリ」にしか見えない。「真のOSを使っている」という認識転換は、UXのどこで起きるべきか。気づかれないまま定着するのが理想か、明示的に提示すべきか。未決定。

### 思想的課題

- **誰のための OS か**: 科学者向けと一般ユーザー向けは UI も優先機能も違う。最初の 1 年でターゲットを決める必要がある。
- **ガバナンス**: オープンソースか商用か。コミュニティの作り方。
- **失敗への態度**: ブラウザベンダーのバグ、Wasm ランタイムの脆弱性に出会ったときの伝播阻止と復旧。Erlang の “let it crash” 哲学のような設計指針が必要。

## 先行研究との関係

本プロジェクトは独自の発明ではなく、既存の思想とプロジェクトの組み合わせである。以下に関係を整理する。

### 個別の先行思想

- **Plan 9 / Inferno**: すべてはファイル、ネットワーク透過
- **Smalltalk / Self**: イメージベース永続化
- **Chromium OS**: ブラウザを OS の中心に据える先行例 (ただし Linux の上のブラウザ、という構図は維持されている)
- **Qubes OS**: VM ベースのサンドボックス分離
- **WebVM / v86**: ブラウザ内 x86 Linux 実行
- **WASIX**: POSIX 互換に寄った WASI 拡張
- **Cosmopolitan libc**: 一つのバイナリで複数 OS

### 本プロジェクトの位置づけ

これらを WASI を共通契約として一本に統合し、以下の組み合わせを試みる:

1. **「ブラウザこそが真の OS」を中心命題として明示する** (原理 1)
   Chromium OS は「Linux の上のブラウザ」という構図を維持していた。本プロジェクトは「ブラウザが OS、ホスト OS は外部資源」という反転を、設計の出発点に置く。
1. **過去の x86 バイナリと未来の WASI ネイティブアプリを同一基盤で扱う**
   WebVM 系は過去の救済、WASIX 系は未来の構築に焦点を絞っている。本プロジェクトは両者を同じシェル、同じファイルシステム、同じ UI で統合する。
1. **タブ複製とスナップショットによるセーブポイント fork**
   WASIX の `proc_fork` は単一インスタンス内の機構として実装されているが、本プロジェクトはブラウザのタブそのものをプロセス的に扱う設計を採る。
1. **AI 砂場プロセスの並走による権限分離**
   Qubes OS の VM ベース分離を、Wasm レベルの軽量な機構で実現する。
1. **WASI シリコンへの段階的移行を視野に入れた設計**
   ソフトウェアで普及させ、普及してからハードウェアに降ろすという順序を、最初から設計思想に組み込む。

### 独自性についての但し書き

調査範囲では、上記の組み合わせを明示的に書いた設計文書は見つけられなかった。しかし「世界初」を主張するつもりはない。同じ発想に至っている人や、未公開の社内プロジェクトは存在する可能性が十分ある。

## 用語集

- **ブラウザ OS**: 本プロジェクトにおいて「真の OS」として扱うブラウザ環境。Chrome / Firefox / Safari の中身。
- **ホスト OS**: ブラウザ OS の下で動く Linux / macOS / Windows / うっすい OS 等。本プロジェクトでは「外部ハードウェア層」と「外部ネットワーク資源を持つ遠隔サーバー」とみなす。
- **WebAssembly (Wasm)**: ブラウザ OS のネイティブ命令セット。CPU アーキテクチャに依存しない。
- **WASI (WebAssembly System Interface)**: Wasm がファイル、ネットワーク、時刻などの「OS の機能」を呼び出すための標準インターフェース。POSIX に思想的に近い。
- **WASI Preview 2**: WASI の 2024 年安定版。コンポーネントモデルと統合された、現実的な使用に耐えるバージョン。
- **コンポーネントモデル**: 異なる言語で書かれた Wasm モジュールを、型安全に組み合わせるための仕様。主に言語間 ABI の統一が目的で、POSIX 互換を直接拡張するものではない。
- **OPFS (Origin Private File System)**: ブラウザ OS のプライマリファイルシステム。本プロジェクトでは `/` 直下にマウント。
- **File System Access API**: ブラウザ OS が「外部ネットワーク資源としての」ホスト OS フォルダにアクセスするための API。
- **CheerpX**: Leaning Technologies が開発する、x86 Linux バイナリをブラウザ内で実行できる JIT 翻訳エンジン。
- **WebVM**: CheerpX を使って Debian Linux をブラウザ内で動かしているデモ / プロダクト。
- **Wasmtime**: Bytecode Alliance によるブラウザ外 WASI ランタイム。サーバーサイドや CLI 用途。
- **CHERI**: ハードウェアレベルでメモリ安全を保証する次世代 CPU 拡張仕様。ポインタに境界情報を持たせる。
- **キオスクモード**: ブラウザを全画面でアプリのように起動するモード。`--kiosk` フラグなどで指定。
- **Service Worker**: ブラウザ OS における init プロセス相当。タブが閉じても生存し、ストレージ層と通信を担う。
- **うっすい OS**: ブラウザ OS の下で走る最小限のホスト OS。Buildroot ベースの Linux 等。

## 参考プロジェクト

### 直接的に近いもの

- **WebVM** (https://webvm.io) — Leaning Technologies. CheerpX で Debian をブラウザ内実行。
- **v86** — ブラウザ内の x86 エミュレータ。Wasm に移植版あり。
- **StackBlitz WebContainers** — ブラウザ内で Node.js を完全実行。仮想 FS 永続化あり。
- **Cosmopolitan libc** — 一つのバイナリが多 OS で動く。思想的に近い。

### WASI ランタイム

- **Wasmtime** (https://wasmtime.dev) — Bytecode Alliance のリファレンス実装。
- **Wasmer** — 商用寄りの WASI ランタイム。Wasmer Registry あり。
- **WasmEdge** — クラウドネイティブ寄りの WASI ランタイム。
- **WAMR (WebAssembly Micro Runtime)** — 組み込み向け軽量ランタイム。

### ブラウザ内 WASI シム

- **@bjorn3/browser_wasi_shim** — シンプルで読みやすい実装。学習用に最適。
- **wasi-js** — もう少し本格的なブラウザ内 WASI 実装。

### ハードウェア関連

- **CHERI / Morello** — ARM の研究プロジェクト。ハードウェアメモリ安全の最前線。
- **RISC-V** — オープンな ISA。WASI ネイティブシリコンの土台候補。

### 思想的に参考になるもの

- **Plan 9 / Inferno** — 「すべてはファイル」、ネットワーク透過。Bell Labs。
- **Chromium OS** — ブラウザを OS の中心的存在として据える先行例。
- **Qubes OS** — VM ベースの権限分離。AI 砂場プロセスの参考。
- **Smalltalk / Self** — イメージベースの永続化。タブ閉じる = サスペンドの参考。
